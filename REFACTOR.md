# Plan: xplane Core Internals Rewrite

## TL;DR

Rewrite the core internals (proxy/tracking, resource model, composition instantiation, handler pipeline) to be more readable, generic, and maintainable — while preserving the user-facing composition API. The current design accumulated complexity because it was built around narrow assumptions (all resources have spec/status, static pending data injection, monolithic handler, duplicated logic between handler and simulator).

---

## Phase 1: Resource Model Redesign

**Goal:** A generic Kubernetes resource model that works with ANY object shape — no assumptions about spec/status.

**Core principle:** A read is a read, a write is a write. No whitelists.

**Design: "Desired-first, fallback-to-observed" proxy**

The resource is a single proxy with unified semantics:

| Action | Path exists in desired? | Behavior |
|--------|------------------------|----------|
| **Read** | Yes | Returns the desired value (already concrete, no dependency edge needed) |
| **Read** | No | Returns a tracked ReadProxy from observed state (creates dependency edge if used by another resource) |
| **Write** | — | Always writes to desired |

```ts
const sg = new SecurityGroup(this, 'sg', {
  spec: { forProvider: { groupName: 'my-sg' } }
});

// ✅ Write to desired — updates the desired document
sg.spec.description = "intent driven FTW";

// ✅ Read desired — returns concrete "my-sg", no edge needed
const name = sg.spec.forProvider.groupName;

// ✅ Read observed — status was never written, falls through to observed
// Creates a dependency edge when used in another resource's props
const sgId = sg.status.atProvider.securityGroupId;
```

**Why this is better than the current system:**

1. Reading a value you already set (like `vpc.spec.forProvider.region`) resolves immediately — no dependency edge, no UNRESOLVED sentinel, no extra iteration needed.
2. No whitelist of "desired vs observed" fields — it's determined purely by what you wrote.
3. Post-construction mutation just works — writes always go to desired.
4. Generic over any K8s object shape — `secret.data.password`, `configmap.data.myKey` all work without special-casing.

**Other details:**

- **Constructor props = the desired document.** Any shape, stored as-is. `{ apiVersion, kind, metadata, spec, data, whatever }` — the framework doesn't care what keys exist.
- **Reserved framework properties** (not proxied) — only TWO:
  - `node` — inherited from `Construct`. Provides tree traversal, and `node.addDependency()` for explicit ordering (standard constructs API, same as CDK).
  - `resource` — namespace for framework configuration. e.g., `sg.resource.autoReady = true`. Provides room to grow without polluting the proxy surface.
  - Everything else (`resourceRef`, `isExisting`, `existingRef`, `toDesired()`, `path`) is internal — stored in private fields or WeakMaps, invisible to the user.
- **`metadata.name`** — NOT auto-generated by the framework. Crossplane handles automatic naming when metadata.name is absent. The framework just passes through whatever the user sets (or doesn't set). `Resource.uniqueName()` remains as a utility static method for when users need deterministic names for resource _fields_ (e.g., AWS security group names), not for metadata.name.
- **`this.pipelineContext`** — read-only accessor over the full Crossplane function context.
- Remove ALL of: `_extra`, `_rootTarget`, `_specTarget`, `_statusTarget`, `_metaTarget`, `_desiredMetaKeys`, `setObserved()`, `setObservedFull()` (replaced by a single `hydrateObserved(data)`)

**Files:**

- `packages/core/src/core/resource.ts` — full rewrite
- `packages/core/src/core/composition.ts` — simplify, remove `_existingResources` special-casing

## Phase 2: Proxy System Simplification

**Goal:** A clearer, more predictable proxy system with fewer modes and edge cases.

**Current problems:**

- `createTrackedProxy` has 4 modes: observed/strict, observed/non-strict, desired/write-tracking, desired/auto-create
- `UNRESOLVED` sentinel gets scattered through objects and must be stripped everywhere
- `deepCloneWithTracked` and `resolveTrackedRefs` are workarounds for the proxy not handling shared references cleanly
- Array bracket notation was added as a patch

**New design:**

- **Two proxy types with clear semantics:**
  1. `ReadProxy` — wraps observed data. Property access records dependency paths. Missing paths return `undefined` (no placeholder proxies). Carries source metadata (owner + path) for dependency edge creation.
  2. `WriteProxy` — wraps desired data. When a `ReadProxy` value is assigned, it registers a dependency edge and stores a `Pending(source)` marker (replaces UNRESOLVED symbol with a typed object that carries its source info).
- **Dependency recording moves to assignment time only** — no scanning with `resolveTrackedRefs` after construction
- **No globalThis hacks for tracking** — use WeakMap for proxy metadata instead of symbols on objects
- **Path handling uses a `Path` value object** — handles dots, brackets, arrays uniformly from the start

**Files:**

- `packages/core/src/tracking/proxy.ts` — full rewrite → split into `read-proxy.ts` and `write-proxy.ts`
- `packages/core/src/tracking/types.ts` — simplify, remove UNRESOLVED export (replaced by Pending type)
- `packages/core/src/tracking/dependency-graph.ts` — keep mostly as-is, minor cleanup

## Phase 3: Composition Instantiation

**Goal:** Eliminate static pending state and globalThis bridge. Replace special-cased environment with full context access.

**New approach:**

- Use **`AsyncLocalStorage`** to pass context into the Composition constructor
- Handler calls `compositionContext.run(ctx, () => new UserComposition())`
- `Composition` constructor reads from ALS — no statics, no globalThis, works naturally with bundled copies
- VM sandbox injects ALS into its globals
- **Replace `this.environment` with `this.pipelineContext`** — a read-only accessor over the full Crossplane function context (all context keys, not just the environment one)
- Composition authors read any context key: `this.pipelineContext.get('apiextensions.crossplane.io/environment')`, `this.pipelineContext.get('my-custom-key')`
- Remove the special-cased environment handling — for EnvironmentConfigs, users can use `Resource.fromExistingByName()` natively (since EnvironmentConfig is just a cluster resource)
- **`this.xr` uses the same "desired-first, fallback-to-observed" proxy as resources.** Reading `this.xr.spec.region` falls through to observed. Writing `this.xr.status.myOutput = someResource.status.prop` writes to desired and creates a dependency edge. The pipeline emits the XR's desired `status` as composite status output. This **replaces `setStatusOutput()`** entirely — no callbacks, just natural assignment.

**Files:**

- `packages/core/src/core/composition.ts` — use AsyncLocalStorage, remove statics and globalThis
- `packages/core/src/core/context.ts` — new file, defines `CompositionContext` and ALS instance
- `packages/function/src/handler.ts` — use context.run() instead of setting statics
- `packages/function/src/loader/sandbox.ts` — ensure ALS is available in VM context

## Phase 4: Rendering Pipeline (shared between handler and simulator)

**Goal:** One pipeline, two consumers. Eliminate the duplicated logic.

**New module:** `packages/core/src/pipeline/`

**Pipeline phases:**

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         Crossplane Function Call                             │
│  (RunFunctionRequest: observed XR, observed composed, context, iteration)   │
└─────────────────────────────┬───────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│  CONSTRUCT                                                                  │
│                                                                             │
│  compositionContext.run(ctx, () => new UserComposition())                   │
│                                                                             │
│  Output: Composition instance with resources, dependency edges, desired     │
│          documents, and Pending markers where ReadProxy values were assigned │
└─────────────────────────────┬───────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│  1. HYDRATE                                                                 │
│                                                                             │
│  Feed observed state (from Crossplane) into each resource's observed store  │
│                                                                             │
│  observed composed resources ──► resource.observed                          │
│  required resources (existing) ──► external resource.observed               │
└─────────────────────────────┬───────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│  2. RESOLVE                                                                 │
│                                                                             │
│  Walk dependency edges. For each Pending(source) in a desired document:     │
│  - Look up source resource's observed data at source path                   │
│  - If concrete value exists → replace Pending with the value                │
│  - If not yet available → leave Pending in place                            │
│                                                                             │
│  Pending("vpc", "status.atProvider.vpcId")                                  │
│       │                                                                     │
│       ▼  observed["vpc"].status.atProvider.vpcId = "vpc-123"                │
│       │                                                                     │
│       ▼  subnet.desired.spec.forProvider.vpcId = "vpc-123" ✓                │
└─────────────────────────────┬───────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│  3. SEQUENCE                                                                │
│                                                                             │
│  Topological sort on dependency graph.                                      │
│  Classify each resource:                                                    │
│                                                                             │
│  - EMIT: no remaining Pending markers, all deps satisfied                   │
│  - BLOCKED: still has Pending markers or deps not yet observed              │
│                                                                             │
│  ┌───┐     ┌────────┐     ┌────────┐                                       │
│  │VPC│────▶│ Subnet │────▶│  App   │                                        │
│  └───┘     └────────┘     └────────┘                                        │
│   emit      blocked        blocked                                          │
└─────────────────────────────┬───────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│  4. DIAGNOSE                                                                │
│                                                                             │
│  For each BLOCKED resource, produce structured diagnostics:                 │
│                                                                             │
│  { resource: "subnet",                                                      │
│    pendingPaths: [{                                                          │
│      path: "spec.forProvider.vpcId",                                        │
│      waitingOn: { resource: "vpc", path: "status.atProvider.vpcId" }        │
│    }] }                                                                     │
└─────────────────────────────┬───────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│  5. EMIT                                                                    │
│                                                                             │
│  Serialize each EMIT resource's desired document → plain K8s object         │
│  Strip any framework internals, produce final desired state                 │
│                                                                             │
│  Output: { emitted: KubernetesResource[], blocked: DiagnosticReport[] }     │
└─────────────────────────────┬───────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│  ADAPTER (handler or simulator)                                             │
│                                                                             │
│  Handler: maps to RunFunctionResponse (SDK resources, conditions, context)  │
│  Simulator: maps to Template + DiagnosticReport for test assertions         │
└─────────────────────────────────────────────────────────────────────────────┘
```

1. **Hydrate** — feed observed state from Crossplane into resource read-proxies
2. **Resolve** — walk dependency edges, resolve concrete values from observed data into desired documents where sources are available
3. **Sequence** — topological sort, classify resources as emit/blocked based on whether all Pending markers are resolved. **Circular dependencies** are detected here (not thrown — produced as a diagnostic). A cycle means the composition is fundamentally broken and can never converge.
4. **Diagnose** — for blocked resources, produce structured diagnostics (which paths are pending, what they're waiting on). For cycles, produce a `CircularDependency` diagnostic with the full cycle path and the field-level detail of what causes it. The handler maps this to a fatal XR condition; the Simulator surfaces it in the report for test assertions.
5. **Emit** — serialize emit-set resources to desired Kubernetes objects

Each phase is a pure function: `(state: PipelineState) => PipelineState`

**The handler** becomes: load composition → construct → run pipeline → map to Crossplane SDK response + conditions

**The simulator** becomes: construct → inject mock observed → run pipeline → return Template

**Files:**

- `packages/core/src/pipeline/index.ts` — orchestrator
- `packages/core/src/pipeline/hydrate.ts`
- `packages/core/src/pipeline/resolve.ts`
- `packages/core/src/pipeline/sequence.ts`
- `packages/core/src/pipeline/diagnose.ts`
- `packages/core/src/pipeline/emit.ts`
- `packages/function/src/handler.ts` — thin adapter calling pipeline + Crossplane SDK mapping
- `packages/devtools/src/assertions/simulator.ts` — thin adapter calling pipeline + Template mapping

## Phase 5: Diagnostics & XR Conditions

**Goal:** Clear, actionable information about why composition is blocked.

**Changes:**

- The `diagnose` pipeline phase produces a structured `DiagnosticReport`:
  ```ts
  interface DiagnosticReport {
    resource: string;
    pendingPaths: Array<{
      path: string;
      waitingOn: { resource: string; path: string };
    }>;
  }
  ```
- Handler maps diagnostics to XR conditions with clear messages: _"Resource 'subnet' is waiting for 'vpc' to provide status.atProvider.vpcId"_
- Expose `DiagnosticReport` from `@xplane/core` so the Simulator can also surface it in tests
- Remove the ad-hoc `findUnresolvedPaths` + `blockReasons` logic from handler — replaced by pipeline's diagnose phase

**Files:**

- `packages/core/src/pipeline/diagnose.ts` — structured diagnostic generation
- `packages/function/src/handler.ts` — map diagnostics to conditions/warnings

## Phase 6: Existing Resources Cleanup

**Goal:** Existing resources are just resources with `observed`-only semantics, no special-casing.

**Changes:**

- `Resource.fromExistingByName()` creates a resource marked as `external: true` — it has an `observed` proxy but no `desired` proxy
- Remove separate `_existingResources` map — they live in the same resource registry, filtered by `external` flag
- Required-resources handling stays in the handler (Crossplane-specific), but the resource itself doesn't need to know about iteration mechanics

**Files:**

- `packages/core/src/core/resource.ts` — unified with the new model
- `packages/function/src/handler.ts` — required-resources loop simplified

## Verification

1. **Unit tests pass** — all existing tests in core, function, devtools must pass (update test internals as needed, but assert same behaviors)
2. **Snapshot tests** — add pipeline snapshot tests: given XR + observed state → assert emitted desired resources + diagnostics
3. **Integration test** — run against a real Crossplane function request/response cycle (use existing test fixtures)
4. **API compat check** — ensure the public API surface (`Composition`, `Resource`, `Resource.fromExistingByName`, `this.xr`) remains unchanged. **`setStatusOutput()` is removed** — replaced by `this.xr.status.field = value` assignment.
5. **Simulator parity** — simulator and handler produce identical results for the same inputs
6. **Bundle test** — fully bundled composition still works (AsyncLocalStorage bridge works in VM sandbox)

## Decisions

- **AsyncLocalStorage over statics** — cleaner, no module-identity issues, works with bundles
- **Full rewrite not incremental** — the issues are interconnected (proxy design affects resource model affects handler); piecemeal changes would create more intermediate states
- **Pipeline in core, not function** — so devtools/simulator can use it without depending on @xplane/function
- **Backward-compatible user API** — `this.xr.spec.foo`, `vpc.status.bar`, `Resource.fromExistingByName()` all keep working. `setStatusOutput()` is removed (replaced by direct `this.xr.status` assignment — simpler, more consistent).
- **Scope boundary** — codegen package is NOT touched (it generates types, which will still work with the new Resource generics). Loader system is NOT touched (just the handler adapter around it).
- **File structure is free to change** — rearrange files, rename modules, restructure directories as needed for clarity. No obligation to preserve existing file layout within packages.
- **Typed XR via generics** — `Composition<TSpec, TStatus, TContext>` makes `this.xr` and `this.pipelineContext` fully typed. Codegen already emits Spec/Status interfaces from XRDs — users just pass them: `class MyComposition extends Composition<MyClaimSpec, MyClaimStatus, MyContext> { ... }`. `TContext` defaults to `Record<string, unknown>` when not provided. `this.pipelineContext.get(key)` returns the typed value for that key.

---

## Further Considerations

1. **AsyncLocalStorage in VM sandbox** — The sandbox runs user code in a `vm.Context`. We need to verify that `AsyncLocalStorage` works across the vm boundary, or inject the ALS store reference into the VM globals explicitly. Needs a spike/proof-of-concept early in Phase 3.
2. **Breaking change: `this.environment` → `this.pipelineContext`** — This is a user-facing API change. The `this.pipelineContext` accessor is more general and future-proof, but existing compositions using `this.environment` will need updating. We could keep `this.environment` as a deprecated getter for one release cycle.
